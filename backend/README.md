# backend (com.foodopenlab.api)

개인·실험용 **모노레포 성격의 작업 공간**이다. 하위에는 앱과 문서가 있고, 루트에는 **LLM 코딩 하네스**(규칙·맥락·검증 루프)를 둔다.

**하네스 엔지니어링:** 모델 가중치를 바꾸는 일이 아니라, **도구·지시·맥락·완료 기준**을 설계해 흔한 실패(묵시적 가정, 과한 일반화, 무분별한 diff, “일단 됨” 종료)를 줄이는 일이다. Andrej Karpathy가 네 가지로 정리한 행동 규범을 이 저장소의 기본 하네스로 삼는다.

**트레이드오프:** 속도보다 신중함. 사소한 작업은 판단으로 완화한다.

## Karpathy 의도 (네 가지, 요약)

| 원칙 | 한 줄 |
|------|--------|
| **1. 구현 전 사고** | 가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것. |
| **2. 단순성 우선** | 문제를 푸는 데 필요한 최소한의 코드만. 추측성 확장은 없다. |
| **3. 정밀한 수정** | 꼭 필요한 곳만 건드린다. 정리는 자기가 만든 잔여물만 한다. |
| **4. 목표 중심 실행** | 성공 기준을 정의한다. 검증될 때까지 반복한다. |

전문·예시·템플릿은 [`CLAUDE.md`](CLAUDE.md)에 있다.

## 완료 기준·성공 신호

- “끝났다”는 **명령 실행·테스트 통과·합의된 수동 확인** 중 최소 하나로 설명 가능해야 한다.  
- 하네스가 잘 먹히면: 불필요한 diff가 줄고, 과한 설계로 되돌리는 일이 줄며, **실수를 낸 뒤가 아니라 구현 전에** 질문·대안 제시가 늘어난다.
